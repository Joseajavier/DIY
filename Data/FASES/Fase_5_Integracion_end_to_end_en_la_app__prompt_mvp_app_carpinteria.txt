LO QUE QUIERO QUE IMPLEMENTES

1. Flujo DIY completo en la app
Quiero que el usuario pueda:
- escribir su idea de proyecto
- enviar la solicitud al backend /ai/diy-generate
- ver loading real
- ver errores si algo falla
- reintentar
- ver resultado estructurado en pantalla
- guardar el resultado como proyecto local
- reabrirlo después desde almacenamiento local
- editar nombre o descripción si procede

Pantallas o componentes implicados:
- DIYInputScreen
- pantalla o sección de resultado DIY
- integración con repositorios de Fase 2 v2

2. Flujo PRO completo en la app
Quiero que el usuario pueda:
- introducir o revisar piezas y parámetros
- lanzar el análisis PRO contra /ai/pro-plan
- ver loading real
- ver avisos o warnings
- ver resultado técnico estructurado
- ver resumen de optimización
- ver materiales estimados
- ver recomendación de compra
- guardar el proyecto y sus resultados localmente
- volver a abrirlo después

Pantallas o componentes implicados:
- ProInputScreen
- ProOptimizationScreen
- ProResultsScreen
- integración con storage y cliente API

3. Gestión de estado y UX
Quiero una solución simple y robusta para manejar:
- idle
- loading
- success
- error
- retry

Puedes usar estado local o una solución ligera si ya existe, pero no quiero sobreingeniería.
Si basta con hooks y servicios bien organizados, perfecto.

Quiero también:
- indicadores de carga claros
- errores entendibles por humanos
- botón de reintento
- evitar dobles envíos
- proteger navegación si hay una operación en curso cuando tenga sentido

4. Persistencia útil
Quiero que decidas claramente qué guardar y cuándo.

Ejemplos:
- guardar automáticamente último proyecto abierto
- guardar drafts si tiene sentido
- guardar resultados confirmados
- no guardar basura temporal innecesaria

Quiero que reutilices:
- SQLite para proyectos, piezas, materiales, optimizaciones, tiendas y resultados que merezca conservar
- MMKV para selectedMode, lastProjectId, units, flags ligeros

Si necesitas ampliar ligeramente el modelo de datos para guardar resultado IA o metadata, hazlo con criterio y sin romper Fase 2 v2.

5. Fallback a lógica local
Si lo ves sensato, quiero que implementes un fallback cuando falle el backend o no haya conectividad:
- DIY: usar diyGenerator.ts local como respaldo
- PRO: usar motor local de materiales/optimización cuando sea razonable

Pero:
- no quiero un sistema caótico
- no quiero resultados inconsistentes sin avisar
- si usas fallback, el usuario debe saber que es una estimación local

Elige una estrategia clara y mantenible.

6. Servicios de aplicación

- UI
- cliente API
- storage
- lógica local fallback

Quiero algo como:

- proWorkflowService.ts o useProWorkflow.ts
- mappers si hacen falta
- helpers de guardado si hacen falta

- ResultSection
- WarningList

- MaterialList
