LO QUE QUIERO QUE IMPLEMENTES

1. Entornos y configuración
Quiero separar claramente:
- development
- staging o beta
- production

Necesito:
- variables por entorno
- API base URL por entorno
- flags básicos si hacen falta
- configuración limpia para Expo y backend

Quiero una solución clara y mantenible.
No quiero una jungla de configuraciones.

2. Builds beta reales
Quiero que definas y prepares cómo generar builds para:
- iOS beta
- Android beta

Si encaja bien con Expo:
- EAS Build
- perfiles dev / preview / production

Quiero:
- archivos de configuración necesarios
- estrategia de perfiles
- comandos de build
- qué build usar para pruebas internas

3. Observabilidad mínima
Quiero implementar una base seria de observabilidad.

Como mínimo:
- logging útil en backend
- trazabilidad mínima de errores
- identificar fallos en llamadas a IA
- tiempos básicos de respuesta
- errores de app que merezca registrar

No quiero un monstruo enterprise.
Quiero lo suficiente para no ir ciego en beta.

4. Crash reporting
Quiero que decidas e implementes una solución razonable para capturar errores de la app y, si procede, del backend.

Elige una solución clara y mantenible.
Puede ser, por ejemplo:
- Sentry
o algo similar

Pero no me des tres opciones.
Elige una y sigue.

Quiero:
- instalación
- configuración básica
- ejemplos mínimos de captura
- cuidado para no exponer secretos

5. Analytics de producto
Quiero analytics mínimos, orientados a saber si la app se usa de verdad.

No quiero vanity metrics absurdas.
Quiero eventos útiles como:
- app_opened
- mode_selected
- diy_request_started
- diy_request_succeeded
- diy_request_failed
- pro_request_started
- pro_request_succeeded
- pro_request_failed
- project_saved
- project_reopened
- fallback_used
- recommendation_accepted si aplica

Quiero que elijas una solución razonable y simple.
Puede ser una solución SaaS si tiene sentido, pero elige una y sigue.

6. Feedback de usuarios
Quiero una vía simple para recoger feedback durante beta.

Puede ser:
- botón “Enviar feedback”
- pantalla simple de feedback
- abrir email
- enviar al backend
- o una solución muy ligera pero usable

Elige una estrategia razonable.
No quiero una feature enorme.
Quiero algo que funcione para una beta pequeña.

7. Release checklist
Quiero una checklist operativa real para lanzar beta o preparar publicación.

Debe cubrir como mínimo:
- variables de entorno
- build correcta
- pruebas mínimas
- errores críticos revisados
- endpoints funcionando
- storage validado
- IA validada
- fallback validado
- analytics y crash reporting activos
- revisión de textos visibles
- revisión de permisos
- plan de rollback o reacción ante errores graves

8. Seguridad operativa básica
Quiero asegurar que:
- no se expone la API key en la app
- secrets en backend y CI
- logs no filtran datos sensibles
- el feedback o analytics no manda más datos de los necesarios
- manejo básico de rate limiting y errores del backend

9. Preparación para publicación futura
No quiero que publiques todavía necesariamente, pero sí dejar listo:
- nombre y bundle coherentes
- iconos y splash si procede
- permisos mínimos claros
- metadatos o checklist para store más adelante
- una estrategia razonable para pasar de beta a producción

10. Cierre operativo
Quiero que al final de esta fase el proyecto quede:
- listo para test con usuarios reales
- con visibilidad de errores
- con medición mínima
- con builds repetibles
- con proceso claro de release
