LO QUE QUIERO QUE IMPLEMENTES

1. QA funcional de flujos críticos
Quiero revisar y reforzar como mínimo:
- abrir app
- elegir DIY o PRO
- crear proyecto DIY
- crear proyecto PRO
- guardar proyecto
- reabrir proyecto
- editar proyecto
- relanzar IA
- fallback local
- historial
- borrado / duplicado
- errores de red
- errores de backend
- errores de storage

Quiero que identifiques huecos reales y los cierres con criterio.

2. Manejo de edge cases
Quiero cubrir casos como:
- inputs vacíos
- inputs absurdos
- piezas con medidas inválidas
- tiempos de respuesta largos
- timeouts
- backend caído
- respuesta IA incompleta
- storage corrupto o lectura fallida
- duplicados
- navegación rara entre pantallas
- doble submit
- volver atrás durante operaciones críticas

No quiero teoría.
Quiero solución práctica en app y backend cuando haga falta.

3. Rendimiento
Quiero revisar y mejorar:
- renders innecesarios
- listas de proyectos
- listas de materiales / pasos / piezas
- uso de hooks
- memoización donde tenga sentido
- cargas iniciales
- navegación entre pantallas
- evitar trabajo repetido

No quiero optimización obsesiva.
Quiero mejoras con impacto real y mantenibles.

4. Accesibilidad mínima seria
Quiero dejar cubierto como mínimo:
- tamaños y contraste razonables
- labels o accessibilityLabel en botones clave
- jerarquía visual legible
- estados de error comprensibles
- targets táctiles razonables
- navegación usable sin depender solo del color
- textos suficientemente claros

No quiero una auditoría infinita.
Quiero un nivel serio para no dejar esto descuidado.

5. Hardening visual y de UX
Después del rediseño, quiero revisar:
- consistency de espaciados
- estados de carga
- estados vacíos
- feedback de éxito
- feedback de error
- diálogos de confirmación
- textos visibles al usuario
- casos donde la UI se rompe o queda rara

6. Hardening de backend y llamadas IA
Quiero reforzar:
- timeouts razonables
- retries controlados donde tenga sentido
- errores normalizados
- validación de respuesta
- logs útiles
- evitar que una mala respuesta de IA rompa la app
- límites básicos de coste o uso si procede

7. Test plan real
No necesito una suite gigantesca si no compensa, pero sí quiero:
- estrategia de tests mínima útil
- test plan manual claro
- si ves útil, tests unitarios en piezas críticas
- si ves útil, tests de integración en partes clave

Quiero que elijas una estrategia razonable y pragmática.

8. Release candidate checklist
Quiero una checklist final tipo RC con:
- flujos críticos validados
- errores graves resueltos
- rendimiento aceptable
- crash reporting activo
- analytics activo
- build beta final probada
- textos revisados
- accesibilidad mínima revisada
- backend estable
- fallback razonable funcionando

9. Decisiones técnicas claras
Quiero que tomes decisiones sin ambigüedad sobre:
- qué optimizar y qué no tocar
- qué edge cases cubrir ya y cuáles no merecen la pena ahora
- cuánto testing meter en esta fase
- qué parte del backend endurecer primero
- qué significa “listo para RC”

No me des veinte opciones.
Elige una solución razonable y sigue.
